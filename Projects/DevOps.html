DevSecOps Pipeline

Project by : Abdallah Yasir - Ehab Thair - Mohammad Yonus - Waleed Mohammad

Automated Security Tests For Web Applications That Written With Python

بواسطة : عبدالله ياسر - ايهاب ثائر - محمد يونس - وليد محمد

في بيئة تتغير بسرعة، لم يعد يكفي أن نطور برامج تعمل؛ بل يجب أن نطورها بذكاء، وننشرها بأمان، ونحافظ على استقرارها باستمرار. من هنا، جاء هذا المشروع ليعكس جوهر منهجية DevOps، حيث يتحول العمل من مراحل منفصلة إلى دورة متصلة، يتحرك فيها الكود من الفكرة إلى الإنتاج بسلاسة.

بهدف تحسين سرعة وجودة بناء البرمجيات. فبدل أن يعمل كل فريق بشكل مستقل، يعمل الجميع ضمن نظام واحد متكامل، مما يؤدي إلى:

  • تسريع عملية تطوير البرمجيات
  • تقليل الأخطاء والمشاكل أثناء النشر
  • تحسين القدرة على الاستجابة للتغييرات
  • تقوية أمان النظام من خلال دمج أدوات الفحص والمراقبة في كل مرحلة

في مشروعنا، لم نركّز على الأدوات فقط، بل على الفكر خلفها. صممنا خط تطوير ونشر متكامل (CI/CD) لا يكتفي بالتنفيذ، بل يراقب، يحلل، ويؤمّن. أدرجنا أدوات أمان مبكرة ضمن دورة التطوير (DevSecOps)، فصارت كل مرحلة نقطة دفاع، وكل خطوة مساهمة في جودة التجربة النهائية.

المرحلة الأولى: الفحص الساكن (Static Analysis)

في المرحلة الأولى يتم فحص الكود في حالته الساكنة دون الحاجة إلى تشغيله:

  • فحص للثغرات الشائعة في الكود
  • التحقق من وجود ثغرات في بعض إصدارات المكتبات
  • فحص تنسيق الكود وقواعد كتابته في HTML و CSS
  • التأكد من عدم وجود أي بيانات حساسة مكتوبة بشكل صريح مثل كلمات المرور وغيرها

في هذه المرحلة قمنا بتثبيت عدد من الأدوات بشكل محلي على الحاسوب كما قمنا ببرمجة كود Powershell يعمل على تشغيل هذه الأدوات دفعة واحدة بهدف تسهيل عملية الفحص على المطوّر.

الأدوات المستخدمة في هذه المرحلة هي:

  • Bandit
  • Gitleaks
  • Snyk
  • Pyright
  • Pylint
  • Flake8

أجرينا التجارب على تطبيق مكتوب بلغة بايثون وقد تعمدنا إدراج عدد من الثغرات بداخله بهدف التحقق من فاعلية الأدوات في كشفها.

وكانت الثغرات كما يلي:

  • Broken Access Control
  • XSS
  • Plaintext Password Storage
  • Missing CSRF Protection
  • Weak Session Key
  • No Input Validation

مستودع الجزء المحلي من المشروع

الجزء الثاني من هذا المشروع يتمثل بأنبوب CI/CD حيث قمنا بتعديل الكود الاساسي للتأكد من خلوه من الثغرات حتى يجتاز الفحص بعد رفعه على GitHub.

الهدف من الـ CI/CD

يهدف الـ CI/CD إلى أتمتة جميع العمليات المتعلقة ببناء وتشغيل ونشر التطبيق، من خلال إعداد Workflow مخصص. هذا الـ Workflow مقسم إلى 6 مراحل رئيسية، بحيث يتم تشغيله تلقائيًا عند رفع الكود الخاص بالتطبيق أو موقع الويب (Push to Repository). يقوم الـ Workflow بتنفيذ جميع المهام المتبقية بشكل أوتوماتيكي بدون تدخل يدوي.

1. الفحص الأمني (Security Check)

في هذه المرحلة يتم فحص الكود المصدري للتطبيق بحثًا عن الثغرات الأمنية باستخدام أدوات SAST (Static Application Security Testing). الأدوات المستخدمة:

  • Bandit: أداة لتحليل كود Python فقط وكشف الثغرات الشائعة
  • Gitleaks: لاستخراج أي أسرار أو رموز حساسة مثل كلمات المرور وTokens المخزنة بالخطأ داخل الكود
  • Semgrep: أداة تحليل قوية تدعم عدة لغات، تشبه Bandit لكنها أوسع وأشمل
  • Black: أداة لتنسيق كود Python بشكل قياسي، لضمان توحيد النمط

إضافة إلى هذه الأدوات، يتم استخدام أدوات GitHub المدمجة مثل:

  • Dependabot: يفحص ملفات requirements.txt أو package.json للتحقق من إصدارات المكتبات ويُنبّه حول أي ثغرات معروفة (CVEs)
  • CodeQL: أداة SAST متقدمة مقدمة من GitHub لفحص الكود البرمجي
  • Secret Scanner: لاكتشاف الأسرار مثل كلمات المرور ومفاتيح API تمامًا مثل Gitleaks

بعد تشغيل الفحص، يتم توليد تقارير لكل أداة. هناك شرط (Condition) داخل الـ Workflow يتجاهل النتائج التي تكون Info أو Warnings أو Low Risk، ويركّز فقط على الثغرات High أو Critical. في حال تم اكتشاف أي ثغرة خطيرة، يتم إيقاف الـ Workflow تلقائيًا قبل الدخول في المراحل التالية.

كما تم ربط الـ Workflow مع بوت على Telegram لإرسال إشعارات في حال نجاح أو فشل الفحص، بهدف مراقبة ما يحدث لحظة بلحظة.

2. البناء والرفع (Build and Push)

في هذه المرحلة، يتم بناء التطبيق داخل Docker Image:

  1. أولًا، يتم تسجيل الدخول إلى Docker Hub
  2. ثم يُبنى التطبيق داخل صورة Docker
  3. بعد نجاح البناء، يتم رفع (Push) الصورة إلى الريجستري (Docker Hub)

3. فحص Dockle (Dockle Scan)

يتم فحص صورة Docker باستخدام أداة Dockle:

  • تكشف عن أي إعدادات خاطئة، مشاكل في Permissions، أو ثغرات أمنية داخل الصورة
  • ُولد تقرير شامل ليتم مراجعته يدويًا لاحقًا

4. النشر على بيئة الاختبار (Deploy to Staging)

بعد التأكد من سلامة صورة Docker، يتم نشر التطبيق على بيئة Staging (بيئة شبه وهمية مخصصة للاختبار). هذه الخطوة ليست فقط لاختبار التطبيق، بل تُعتبر أيضًا طبقة إضافية من الأمان قبل النشر النهائي.

5. فحص الويب أثناء التشغيل (Web Scanner - Runtime Scanning)

في هذه المرحلة يتم فحص التطبيق وهو يعمل (Runtime Scan) باستخدام أدوات مثل:

  • Nikto: لفحص التطبيق على مستوى الويب واكتشاف ثغرات مثل XSS، Headers غير آمنة، وغيرها

إذا اجتاز التطبيق هذا الفحص بنجاح ولم تظهر أي مشاكل، ننتقل إلى المرحلة الأخيرة.

6. النشر النهائي (Deploy to Production)

يتم نشر التطبيق على بيئة Production ليكون متاحًا للمستخدمين النهائيين.

مستودع خط الأنابيب الكامل CI/CD

English Version

In a fast-changing environment, it's no longer enough to simply develop functioning software; we must develop it smartly, deploy it securely, and continuously maintain its stability. This project embodies the essence of the DevOps methodology, where work transitions from isolated stages into a continuous cycle—moving code from idea to production seamlessly.

The goal is to enhance both the speed and quality of software development. Rather than having teams work independently, all collaborators function within one integrated system, which leads to:

  • Faster software development
  • Fewer deployment errors
  • Improved adaptability to change
  • Stronger system security through integrated scanning and monitoring tools at every stage

In our project, we focused not only on tools but on the mindset behind them. We built a complete CI/CD pipeline that doesn't just execute but monitors, analyzes, and secures. We integrated security tools early in the development cycle (DevSecOps), making each stage a defensive layer and every step a contribution to the final experience quality.

Phase 1: Static analysis is performed without running the code. This includes vulnerability scans, library version checks, HTML/CSS formatting validation, and detection of hardcoded sensitive data like passwords.

We locally installed multiple tools and created a PowerShell script to run them all together to simplify the process for developers.

Tools Used:

  • Bandit
  • Gitleaks
  • Snyk
  • Pyright
  • Pylint
  • Flake8

We tested the tools on a vulnerable Python app to validate their effectiveness. The vulnerabilities included:

  • Broken Access Control
  • XSS
  • Plaintext Password Storage
  • Missing CSRF Protection
  • Weak Session Key
  • No Input Validation

DevOps Local Part

The second part of the project is a CI/CD pipeline. We modified the code to be vulnerability-free before pushing it to GitHub so it can pass the automated checks.

CI/CD Goals:

To automate the build, run, and deployment of the app using a 6-stage workflow triggered upon code push:

1. Security Check

Performs SAST using:

  • Bandit: Python vulnerability scanner
  • Gitleaks: Detects secrets and tokens in code
  • Semgrep: Multi-language SAST tool
  • Black: Python code formatter

Also uses GitHub-native tools:

  • Dependabot: Scans dependencies for known CVEs
  • CodeQL: GitHub's advanced code analysis tool
  • Secret Scanner: Finds credentials like Gitleaks

Reports are generated for each tool. The workflow filters out low-risk findings and halts if critical/high vulnerabilities are detected. It’s also integrated with a Telegram bot for real-time notifications.

2. Build and Push

Logs in to Docker Hub, builds the app image, and pushes it to the registry. An additional security scan is done next.

3. Dockle Scan

Scans Docker image for misconfigurations, permission issues, and embedded vulnerabilities.

4. Deploy to Staging

The image is deployed to a testing environment (staging) to ensure stability before public release.

5. Web Scanner (Runtime Scanning)

Scans the live app using tools like Nikto to detect web-based vulnerabilities like XSS, unsafe headers, etc.

6. Deploy to Production

The secure app is deployed for end users.
